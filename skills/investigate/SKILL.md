# Sentry 이슈 조사

사용자가 설명한 이슈를 Sentry에서 찾아 근본 원인을 분석합니다.

## 입력

$ARGUMENTS

## 실행 절차

아래 단계를 **순서대로** 수행하세요. 각 단계의 결과를 다음 단계에서 활용합니다.

---

### 1단계: 입력 파싱

사용자 입력에서 다음을 추출합니다:
- **이슈 키워드**: 에러 메시지, 기능명, 증상 등 (예: "결제 실패", "500 error", "timeout")
- **날짜/시간대** (있으면): 발생 시점 (예: "3월 18일", "어제 오후 3시~5시")
- **프로젝트** (있으면): 특정 앱 지정 (예: "백엔드", "커머스", "모바일")

날짜가 없으면 최근 24시간으로 기본 설정합니다.

### 2단계: 프로젝트/조직 확인

Sentry MCP 도구를 사용합니다:

```
find_organizations → 조직 목록 확인
find_projects → 프로젝트 목록 확인 (사용자가 지정한 앱이 있으면 해당 프로젝트 선택)
```

### 3단계: 이슈 검색

자연어 검색으로 관련 이슈를 찾습니다:

```
search_issues → 키워드 기반으로 관련 이슈 검색
```

- 검색어는 사용자 입력의 핵심 키워드를 사용
- 결과가 너무 많으면 시간대, 프로젝트로 필터링
- 결과가 없으면 키워드를 변형하여 재검색 (영어/한글 변환, 유사어 시도)

### 4단계: 이슈 상세 분석

가장 관련성 높은 이슈(들)의 상세 정보를 조회합니다:

```
get_issue_details → 스택트레이스, breadcrumbs, 태그, 발생 횟수, 영향 사용자 수
get_issue_tag_values → 환경(environment), 브라우저, OS 등 분포
search_issue_events → 해당 이슈의 개별 이벤트 상세 (필요 시)
```

**확인해야 할 핵심 정보:**
- 스택트레이스 (에러 발생 위치)
- Breadcrumbs (에러 발생 전 사용자/시스템 행동 흐름)
- 태그 (environment, browser, os, user 등)
- 발생 빈도 및 추세 (최근 급증 여부)

### 5단계: Sentry AI 근본 원인 분석 (선택)

Sentry의 Seer AI가 사용 가능한 경우 활용합니다:

```
analyze_issue_with_seer → AI 기반 근본 원인 분석 및 코드 수정 제안
```

### 6단계: 코드베이스 분석

Sentry에서 얻은 정보를 기반으로 실제 코드를 분석합니다:

**스택트레이스 기반:**
- 에러가 발생한 파일명/함수명을 `Grep`으로 코드베이스에서 검색
- 해당 코드를 `Read`로 읽어 로직 분석

**에러 메시지 기반:**
- 에러 메시지 문자열을 `Grep`으로 검색하여 throw/raise 지점 확인
- 에러 클래스 정의 및 사용처 확인

**API 엔드포인트 기반:**
- breadcrumbs에서 나온 API 경로를 라우팅 코드에서 추적
- 해당 핸들러/서비스 로직 분석

**외부 의존성 기반:**
- 외부 API 호출 실패인 경우 타임아웃/재시도 로직 확인
- DB 쿼리 실패인 경우 스키마/인덱스 확인

### 7단계: 종합 보고서 생성

아래 형식으로 결과를 보고합니다:

```markdown
## Sentry 이슈 조사 보고서

### 이슈 요약
- **이슈**: [이슈 제목] (Sentry ID: XXXX)
- **최초 발생**: [날짜]
- **발생 횟수**: [N]회 / 영향 사용자: [N]명
- **환경**: [production/staging]
- **상태**: [unresolved/resolved/ignored]

### 타임라인
1. [시간] — [이벤트 설명]
2. [시간] — [이벤트 설명]

### 근본 원인 분석
[스택트레이스 + 코드 분석을 종합한 원인 설명]

**에러 발생 위치:**
- 파일: `path/to/file.ts:line`
- 함수: `functionName()`

**원인:**
[구체적인 원인 설명]

### 재발 방지 제안
1. [제안 1]
2. [제안 2]

### 관련 코드
[핵심 코드 스니펫 및 수정 제안]
```

---

## 주의사항

- Sentry MCP 도구 호출이 실패하면 사용자에게 알리고 OAuth 인증 상태를 확인하도록 안내
- 여러 이슈가 발견되면 가장 관련성 높은 것부터 분석 (최근 발생, 높은 빈도 우선)
- 코드 분석 시 현재 워킹 디렉토리의 코드베이스를 사용
- 프로덕션과 스테이징 환경을 구분하여 분석
